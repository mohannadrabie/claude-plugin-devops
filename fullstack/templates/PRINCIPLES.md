# Working Charter — how this team works (anti-kafka rules)
The point of governance is confidence, not ceremony. These rules bind agents AND humans.

1. **Ceremony scales with risk.** Trivial change (docs, tags, comments): tests + self-review, ship. Standard change: ONE domain reviewer (the right one) + implementer. Critical change (guarded paths, prod-facing, security-relevant): red team + the relevant reviewer. Every non-trivial review also seats `fullspectrum-reviewer` — the standing cross-domain pass (rule 9) — but it doesn't count against the cap below: it isn't a domain pick, it's the seam-check the Manager used to reconstruct ad hoc, now done by a receipted specialist. Never more than two *domain* reviewers unless the manager orders it, in writing, with a reason.
2. **Every block names its unlock.** A gate that says "no" without saying "here is exactly how to proceed" is a defect — file it like one.
3. **Approvals expire; work doesn't wait.** Review reports are keys with dates. Deadlocked reviewers escalate to /manager THE SAME DAY — decisions in hours, not standups.
4. **SURVIVES is a celebrated verdict.** Reviewers are not paid in findings. A clean pass, verified, is the goal — manufactured findings destroy auditor credibility scores.
5. **Findings are fixes, not blame.** Every finding is phrased as the minimal change that resolves it. No "who did this"; git remembers so people don't have to.
6. **Auditors get audited — and audits are discipline, not compliance.** /audit-reviewers samples past reports monthly: evidence quality, false positives, softened findings, missed blockers. Reviewer credibility is a tracked score. The manager's decisions are in the sample too. There is no external body being satisfied here — the only question an audit answers is whether the work actually happened. An audit that rubber-stamps a claim it never checked protects nobody; it is worse than not auditing at all, because it hides the gap instead of showing it.
7. **The manager decides, the log remembers, the human overrides.** Manager decisions are logged with rationale + dissent recorded + a review-back date. Nothing is quietly final.
8. **One next step.** Any agent ending a turn must state the single most important next action. Nobody should ever wonder "what now?"
9. **ADRs are the architecture source of truth, and they outrank conventions.** Accepted ADRs — their `Rules for agents` (testable MUST/SHOULD) — are hard constraints ranked ABOVE CLAUDE.md's conventions. Convention vs ADR → the ADR wins; ADR vs ADR → the more specific wins. A rule you cannot satisfy is never silently violated: say so and propose an ADR change (`/…:adr-amend`). Never edit an accepted ADR's decision — supersede it with a new ADR and mark the old one `superseded`. When ADRs are adopted or amended, existing code is realigned through a **conformance triage**, not silent drift. Architecture decisions live only in ADRs — `docs/decisions.md` is the general Decision Log, not an ADR store. **Each domain reviewer reads only its own domain's slice of the ADR catalog** (filter `adrCatalog.adrs` by `applicableTo`) — a data-reviewer reads the data ADRs, not the network ones. **`fullspectrum-reviewer` owns cross-domain collision detection and inter-lane gaps, as a real reviewer with its own dated report and receipt, not an ad hoc Manager read**: it runs on **every review path above TRIVIAL** — including the inline single-domain-reviewer path — reads the compressed catalog **once, whole and unfiltered** (no `applicableTo` slice), and checks the **diff — the changed files — against every applicable ADR outside the lane(s) the domain reviewer(s) covered**, plus hunts non-ADR functional/architectural gaps at the seams between the domains the diff touches. It checks the code, not the other reviewers' findings: a violation in a non-owning lane produces no finding there, so a findings-only check would miss it. This is the single-read design — `1 × whole catalog + N × slice` — deliberately chosen over `N × whole catalog`, because the per-review cost of every reviewer reading everything grows with the ADR corpus while the one compressed read does not. `fullspectrum-reviewer` doesn't count against rule 1's two-domain-reviewer cap — it's the standing seam-check, not a domain pick. `/ship-check` re-runs a whole-catalog-vs-diff ADR pass as the pre-merge backstop — so reviewer-slicing never lets an out-of-lane ADR violation reach merge even if a run somehow skipped it.
10. **Evidence, not claims.** A review, a check, or a "done" is real only when its artifact is committed. Reviewers save their report **verbatim** to `docs/reviews/<scope>-<type>-<YYYY-MM-DD>.md` with the **raw command/test output** (passed/failed/**skipped** counts). A review that lives only in a chat transcript is a claim, not evidence, and opens no gate. "Skipped" is never "passed." Every report-producing agent also ends its turn with a compact `RECEIPT:` block — this is how it hands its verdict to the Manager cheaply, and it is a pointer to the evidence, never a substitute for it. Reviewers (and redteam/challenger) list **every** finding, one terse line each — `[ISSUE]`/`[SUSPICION]`/`[CLEAN]` status, a `[HIGH|MED|LOW]` severity on each issue/suspicion, and a `counts` line that is a checksum on the list (not a top-N — nothing is hidden by length, so there is no `counts>N` trigger). Build and debug receipts carry deterministic counts instead (criteria mapped, checks passed/failed/skipped) since there's nothing to tag. The Manager reopens the actual persisted report — this trigger list is identical in `manager.md` and `ship.md` — the moment any of: the receipt is missing/malformed; `counts` ≠ the listed lines; the verdict isn't that agent's clean value (`manager-summary-format.md` has the per-agent vocabulary table); the verdict contradicts the findings (clean verdict with any `[ISSUE]`, or any `[HIGH]` under a clean verdict); any `[HIGH]` or any `[SUSPICION]`; a terse line reads worse than its severity tag; zero findings on a non-trivial diff; a failed/skipped check under a claimed-clean verdict; or an ADR violation is claimed — and reopens **every** report regardless on a CRITICAL-tier run. The cheap path only ever greenlights (trust-and-relay); a blocker, rework, or re-tier is declared only after a full read. Audit depth scales with blast radius, same as reviewer selection does.
11. **Reports are immutable; verification is appended.** A persisted report is not edited after the fact — a finding that is later verified, disputed, or overtaken gets an **addendum** (or a follow-up dated report); the original evidence stays verbatim. High-severity findings are adversarially verified *before* they gate.
12. **Not in scope → backlog, never code.** Anything worth doing but outside the approved change goes to `docs/backlog.md` (with its origin, and a date if time-bound) — never into the diff as gold-plating. The backlog is where good ideas wait; the diff is only the approved change.
13. **Never skip a gate silently.** If a required gate (review, ADR check, DoD item, human approval) was skipped or could not run, STOP and say so plainly. A silently-skipped gate is the worst failure mode — worse than a failed one, because it hides.
14. **One state, read first.** Every session hydrates the same context, in order — `docs/STATE.md` (where things stand) → `CLAUDE.md` (hard rules) → this charter → `docs/decisions.md` (the **active** set only — resolved rows past their review-back date live in `docs/decisions-archive.md`, read on demand, so this hydration stays bounded as the project ages) → the applicable ADRs — before acting. The manager keeps `docs/STATE.md` current at every close so the loop resumes across sessions.
